# PLP development cycle time

A delivery team used GitHub Copilot between September and November 2025. They measured the impact on development speed and documentation quality. The team saw a 61% reduction in cycle time.

## Overview

The PLP delivery team introduced GitHub Copilot into their workflow between September and November 2025. They wanted to understand whether AI coding assistants could meaningfully improve delivery speed and code quality.

They approached adoption systematically. They provided training to encourage safe and effective use, measured usage patterns, and tracked the impact on delivery metrics.

The results were significant. Development cycle times dropped by 61%, and half of all new feature development included documentation generated with Copilot support.

The adoption process was not straightforward. There was initial resistance and uneven usage patterns. However, the team demonstrated that AI coding assistants could deliver measurable improvements when used effectively.

## How AI supported the work

**Code generation** handled boilerplate particularly well. Copilot generated repetitive patterns quickly. This allowed developers to focus on business logic rather than syntax.

**Documentation** was written at the point of development rather than being left until later. Developers used Copilot to generate inline comments, function documentation, and architecture documentation.

**Pipeline automation** benefited from Copilot handling repetitive configuration code. This freed up developer time for more complex problem-solving work.

**Architecture documentation** became easier to maintain. Copilot helped structure explanations and generate initial drafts that developers could refine.

## Tools used

- GitHub Copilot

## Outcomes and impact

### 61% reduction in development cycle time

Development cycle time dropped by 61% over the three months. The reduction came from faster code writing and better documentation practices.

### 50% of new features included AI-assisted documentation

Documentation coverage improved measurably. Half of all new features included Copilot-generated or Copilot-assisted documentation, compared to lower rates before the trial.

### Faster code review cycles

Well-documented code was easier to review. The team reported that code review cycles became faster because reviewers could understand code more quickly.

### Usage spiked after training

Copilot usage increased sharply after training sessions. This showed developers understood the value once they saw it demonstrated. However, usage declined over subsequent weeks without ongoing support.

## Challenges and what the team learned

**Initial resistance** came from concerns about code quality and whether AI was appropriate for government work. Training that explained how Copilot worked and what safeguards were in place helped address these concerns.

**Competing priorities** deprioritised learning time. Delivery pressure meant developers focused on immediate tasks rather than experimenting with new tools. The team recognised that sustainable adoption required protected time for learning.

**Access to premium features** was inconsistent across the team. This created frustration when some developers saw colleagues benefiting from features they could not access.

### Key lessons

**Training alone is not sufficient.** Teams need ongoing support, not just one-off sessions. Regular check-ins, shared learning sessions, and visible champions made more difference than training alone.

**Clear guidance on basic functionality is essential.** Do not assume users will discover features themselves.

**Measuring impact helped build support.** The 61% reduction in cycle time made it easier to justify continued investment and overcome initial scepticism.
